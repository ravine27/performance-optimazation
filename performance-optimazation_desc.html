Lazy Loading Techniques

Lazy loading delays the loading of non-critical resources (like images, videos, and iframes) until they're needed. Instead of loading everything when the page first opens, resources load as the user scrolls near them. This dramatically reduces initial page weight and improves loading speed.

Native Lazy Loading Implementation

Supported in 90%+ Browsers

The simplest way to implement lazy loading is using the native loading="lazy" attribute:

<!-- Traditional image loading (immediate) --> <img src="hero-banner.jpg" alt="Main Hero Banner"> <!-- Lazy loaded images (load when scrolled near) --> <img src="product-image.jpg" alt="Product" loading="lazy"> <img src="gallery-1.jpg" alt="Gallery Image 1" loading="lazy"> <img src="gallery-2.jpg" alt="Gallery Image 2" loading="lazy"> <!-- Lazy loading for iframes --> <iframe src="video-player.html" loading="lazy"></iframe> <iframe src="embedded-map.html" loading="lazy"></iframe>

Loading on Demand

How Lazy Loading Works:

  • Initial Load: Only above-the-fold images load immediately
  • Scroll Detection: Browser monitors scroll position
  • On-Demand Loading: Images load when user scrolls near them
  • Bandwidth Saving: Mobile users save data automatically
  • Performance Boost: Improves Largest Contentful Paint (LCP)
When to Use Lazy Loading vs Eager Loading

Use loading="lazy" for: Images below the fold, gallery images, product thumbnails, non-critical content

Use loading="eager" (or omit) for: Above-the-fold hero images, logos, critical interface elements, images needed for initial rendering

Resource Hints & Preloading

Resource hints are HTML directives that tell the browser about important resources before they're explicitly requested. By giving the browser advance notice, you can eliminate delays caused by DNS lookups, connection establishment, and resource discovery.

Without Resource Hints

  • Browser discovers resources only when parsing HTML
  • Sequential DNS lookup for each new domain
  • Connection handshake delays for external resources
  • Critical resources compete for bandwidth
Result: Slower initial render, delayed interactive time

With Resource Hints

  • Browser prepares connections in advance
  • Parallel DNS resolution and connection setup
  • Critical resources prioritized and preloaded
  • Next-page resources fetched proactively
Result: Faster loading, better user experience

Complete Resource Hints Implementation

<!-- DNS Prefetching: Resolve domain names early --> <link rel="dns-prefetch" href="//fonts.googleapis.com"> <link rel="dns-prefetch" href="//cdn.example.com"> <!-- Preconnect: Establish full connection to critical origins --> <link rel="preconnect" href="https://api.example.com" crossorigin> <link rel="preconnect" href="https://static.example.com"> <!-- Preload: Force early fetch of critical resources --> <link rel="preload" href="/css/critical.css" as="style"> <link rel="preload" href="/images/hero-image.webp" as="image" type="image/webp"> <link rel="preload" href="/js/main.js" as="script"> <link rel="preload" href="/fonts/roboto.woff2" as="font" type="font/woff2" crossorigin> <!-- Prefetch: Load resources for future navigation --> <link rel="prefetch" href="/next-page.html" as="document"> <link rel="prefetch" href="/products/data.json" as="fetch">

JavaScript Loading Strategies

JavaScript execution blocks HTML parsing by default. Optimizing how and when scripts load is crucial for performance. Modern browsers provide several attributes to control script loading behavior.

Script Loading Attributes Explained

<!-- Default: Render-blocking (bad for performance) --> <script src="app.js"></script> <!-- Stops HTML parsing until loaded & executed --> <!-- ========== OPTIMIZED APPROACHES ========== --> <!-- DEFER: Load in background, execute after HTML parsed --> <script src="analytics.js" defer></script> <!-- Maintains order, good for dependencies --> <!-- ASYNC: Load & execute as soon as available, order not guaranteed --> <script src="widget.js" async></script> <!-- Independent scripts only --> <!-- MODULE: ES6 modules, automatically deferred --> <script type="module" src="main.js"></script> <!-- Modern approach --> <!-- Module with fallback for older browsers --> <script type="module" src="app.js"></script> <script nomodule src="legacy-app.js" defer></script>

Critical Warning About Async Scripts

Async scripts execute as soon as they load, regardless of order! If script B loads before script A, it will execute first. This can break dependencies. Use async only for:

  • Third-party analytics (Google Analytics, Facebook Pixel)
  • Independent widgets (social media buttons, chat widgets)
  • Scripts with no dependencies

For most application scripts, use defer instead!

8 Essential Web Optimization Techniques

Performance Optimization Checklist
1
Minimize HTTP Requests

80% of load time is spent downloading page components. Combine files, use CSS sprites, and eliminate unnecessary resources.

// Before: 10 separate requests <link rel="stylesheet" href="style1.css"> <link rel="stylesheet" href="style2.css"> <script src="script1.js"></script> <script src="script2.js"></script> // After: 2 requests (bundled) <link rel="stylesheet" href="bundle.css"> <script src="bundle.js"></script>
2
Enable Compression (GZIP/Brotli)

Google estimates 99 years of human time wasted daily due to uncompressed content. Enable compression at server level.

3
Use CDNs (Content Delivery Networks)

CDNs distribute content globally, reducing latency. Netflix uses CDNs to cache videos locally for faster streaming.

4
Optimize Images

Images account for 66% of page weight. Use WebP format, responsive images, and proper compression.

5
Implement Caching

Twitter uses Redis to cache timelines and sessions, handling millions of queries per second.

6
Minify CSS, JavaScript, HTML

Remove unnecessary characters (whitespace, comments) without changing functionality.

7
Use Lazy Loading

Load resources only when needed. Pinterest reduced perceived wait times by 40% using lazy loading.

8
Optimize Database Queries

Use indexes, avoid N+1 queries, and implement query caching. Facebook uses Memcached to cache social graph data.

Common Performance Mistakes to Avoid

  • Not compressing images before upload
  • Using too many third-party scripts
  • Loading all JavaScript in the <head>
  • Not setting cache headers
  • Using redirects unnecessarily
  • Serving oversized images to mobile devices

    System Design & Scalability

    Scalability Strategies

    Horizontal Scaling

    Adding more nodes/machines. Netflix uses this approach, spinning up thousands of VMs within minutes during traffic spikes.

    Cost-effective for cloud environments

    Vertical Scaling

    Upgrading existing resources (CPU, RAM). Has physical limits but good for immediate needs.

    Limited by hardware constraints

    Load Balancing

    Distributing traffic across servers. Round-robin, least-connections, and IP-hash strategies.

    Essential for high availability
    Real-World Example: Netflix

    Performance Architecture: Netflix runs over 700 microservices with anticipatory scaling. They dynamically adjust service clusters based on real-time CPU, network, and queue metrics.

    Microservices

    700+ independent services enabling thousands of daily deployments

    CDN Strategy

    Local caching of videos reduces latency and bandwidth costs

    Auto-scaling

    Spin up/down VMs within minutes based on demand
